# Prompt Analysis

### Design Strategy
1. **XML-Style Structuring:**  I used <tag> delimiters throughout the prompt because modern LLMs follow instructions more strictly when content is logically separated. This prevents "instruction leakage" where the bot might forget safety rules during long conversations. The hierarchy—from <role> to <personality> to <university_context>—creates clear containers that help the model distinguish between identity, behavior guidelines, and factual knowledge about campus life.
2. The <situational_responses> section functions as an IF/THEN framework for different user types. Whether someone is a buyer asking about an item, a seller posting their first listing, or a new user needing onboarding, the bot has clear behavioral templates. This ensures consistency while allowing flexibility—the scenarios provide principles, not rigid scripts, so the model can adapt to novel situations while staying aligned with our values.
3. **University-Specific Safety:** Unlike generic marketplaces (e.g., FB Marketplace), a campus environment has high stakes for "Academic Integrity." I prioritized the ban on selling exam/homework materials to align with University code-of-conduct.The escalation protocol with clear triggers (self-harm, threats, prohibited items) ensures high-risk situations reach human moderators immediately, while the academic calendar integration enables seasonally appropriate advice throughout the full year. 
4. **Student Budget Awareness:** I added a concise price guidance section because financial constraints shape student marketplace behavior. Unlike general platforms, students time purchases around semesters and look for deals. The feature provides simple price tiers, seasonal timing advice, and negotiation tips—helping budget-conscious users without overcomplicating the prompt or claiming real-time data access.

### Target Performance
The prompt is designed to minimize hallucinations by grounding responses in the provided university context and providing clear escalation pathways for any scenario outside basic transactions. The tiered enforcement actions (warning → temporary restriction → suspension) create predictable consequences, while the boundary handling templates give the bot graceful ways to say "no" when users request inappropriate information. This structure aims to resolve 80% of queries autonomously while ensuring the 20% involving safety, disputes, or complex issues reach human moderators with all necessary context.
